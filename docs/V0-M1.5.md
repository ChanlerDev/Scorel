# V0-M1.5: Anthropic Adapter

> Anthropic Messages adapter + transformMessages() cross-provider pipeline + tool_call_id normalization

## Goal

Add Anthropic Messages as a second provider. Validate that the canonical message model + adapter abstraction works for a structurally different API.

## Scope

- **Anthropic adapter**: convert canonical messages to/from Anthropic Messages format
- **transformMessages() pipeline**: unified entry point for provider-specific message transformation
- **System prompt extraction**: move system/developer content to Anthropic's top-level `system` parameter
- **Tool result regrouping**: canonical `role: "toolResult"` → `tool_result` blocks inside `role: "user"` messages
- **Tool result ordering**: within user messages, `tool_result` blocks before freeform text
- **Message alternation**: enforce strict user/assistant alternation for Anthropic
- **Tool call ID normalization**: deterministic shortening for IDs > 64 chars (Anthropic limit)
- **Thinking block handling**: preserve `ThinkingPart` for same-model roundtrip; convert to text for cross-provider
- **Stop reason mapping**: `end_turn` → `stop`, `max_tokens` → `length`, `tool_use` → `toolUse`
- **Anthropic SSE normalization**: `message_start/content_block_start/content_block_delta/content_block_stop/message_delta/message_stop` → canonical `AssistantMessageEvent`

## Out of Scope (M1.5)

- Extended thinking configuration UI (V0 does not request thinking by default, but parses/persists it if returned)
- Anthropic-specific streaming options (prompt caching, etc.) — reserve for Beta
- Cross-provider session switching mid-session (sessions are single-provider in V0)

## Key Implementation Notes

### transformMessages(api, messages)

Pure function. Two responsibilities:

1. **Validate + clean**: exclude aborted assistants, exclude orphan tool results, apply micro_compact placeholders
2. **Transform to target format**: OpenAI or Anthropic specific structure

```ts
// Conceptual signature
function transformMessages(
  api: Api,
  messages: ScorelMessage[],
  compat?: ProviderCompat
): ProviderPayload;
```

### Anthropic Ordering Rules

Per COMPAT.md §2, the adapter MUST enforce:

1. No `role: "system"` in messages[] → extract to top-level `system`
2. `tool_result` blocks inside `role: "user"` messages (not standalone)
3. `tool_result` blocks before text blocks within same user message
4. `tool_result` immediately follows corresponding `tool_use` turn
5. Tool call IDs ≤ 64 chars (shorten with `tc_` + SHA-256 prefix)
6. Strict user/assistant alternation (merge or bridge)

### Tool Call ID Normalization

```ts
function normalizeToolCallId(id: string): string {
  if (id.length <= 64) return id;
  const hash = sha256(id).slice(0, 61);
  return `tc_${hash}`; // exactly 64 chars
}
```

- Mapping is adapter-local (not stored)
- Same mapping applied to both outbound `tool_use.id` and `tool_result.tool_use_id`
- Canonical storage keeps original IDs unchanged

### Anthropic SSE → AssistantMessageEvent

| Anthropic SSE | Canonical Event |
|---------------|----------------|
| `message_start` | `start` |
| `content_block_start` (type=text) | `text_start` |
| `content_block_delta` (type=text_delta) | `text_delta` |
| `content_block_stop` (text) | `text_end` |
| `content_block_start` (type=tool_use) | `toolcall_start` |
| `content_block_delta` (type=input_json_delta) | `toolcall_delta` |
| `content_block_stop` (tool_use) | `toolcall_end` |
| `content_block_start` (type=thinking) | `thinking_start` |
| `content_block_delta` (type=thinking_delta) | `thinking_delta` |
| `content_block_stop` (thinking) | `thinking_end` |
| `message_delta` (stop_reason) | (used to finalize) |
| `message_stop` | `done` |

### Cross-Provider Thinking Block Handling

```
Same provider + model → preserve ThinkingPart (including thinkingSignature for cache)
Different provider or model → convert ThinkingPart to TextPart: { type: "text", text: thinking }
Aborted/errored assistant → skip entirely (don't send to any provider)
```
